# MESHGRAM Relay — детальная документация

Эта папка — основной модуль проекта. Здесь живёт сам бот, GUI, конфиг и БД. Документ ниже описывает, **что уже сделано, что планируется и почему мы выбрали именно такие решения**.

Краткий обзор проекта в целом — в [главном README](../README.md) корня репозитория.

## О чём проект

**Meshgram** — личный мост между Telegram и Meshtastic LoRa-сетью. Идея простая: у тебя есть карманная LoRa-нода, и ты хочешь чтобы знакомые могли с тобой связаться, даже когда у тебя нет интернета — лишь бы карманная нода была в зоне приёма твоей mesh-сети.

Архитектура — две точки на радиоканале:

```
        ┌──────────────────┐                                        ┌─────────────────┐
        │  Любой TG-юзер   │                                        │   Получатель    │
        │  (бот в TG)      │                                        │   (карманная    │
        └────────┬─────────┘                                        │   нода LoRa)    │
                 │  «привет, ты где?»                               └────────┬────────┘
                 ▼                                                           ▲
        ┌──────────────────┐                                                 │
        │  relay.py        │  Meshtastic DM (E2E-шифр)                       │
        │  + home node     │ ─────────────────────────────────────────────►  │
        │  (USB)           │  «[@3 vasya 16:15] привет, ты где?»             │
        │                  │                                                 │
        │                  │  ◄───────────────────────────────────────────── │
        └──────────────────┘  «@3 в кафе на ленина»                          │
                 │                                                            │
                 ▼                                                            │
        TG: «в кафе на ленина»                                                │
```

Ключевые свойства:
- **Адресная маршрутизация**: одно и то же сообщение от Васи и от Маши попадают в **разные слоты** `@N`, ответ на конкретный `@N` уходит конкретному человеку. Никакого «общего чата».
- **Sticky-слоты**: повторные сообщения от того же юзера падают в тот же `@N` — у тебя в кармане не плодится бесконечная нумерация.
- **DM, не broadcast**: между двумя своими нодами идёт зашифрованный DM. Чужие узлы на канале только ретранслируют зашифрованный блоб — не видят содержимого.
- **Подтверждения доставки**: TG-пользователь видит «📨 передаю» → «📨 отправлено» → «✓ доставлено» как в обычном мессенджере.
- **Без интернета у получателя** — ему достаточно радиоканала.

| Уровень | Что делает |
|---|---|
| **Mesh-слой** (`relay.py` → `_connect_mesh`, `on_mesh_receive`, `send_dm_to_pocket`) | Подключение к home-ноде по USB, приём пакетов от карманной, отправка DM с ACK |
| **Парсер мешевого текста** | Категории сообщений от карманной ноды: `#SOS`, `!cmd`, `@N text`, `@N !cmd`, raw |
| **Слот-менеджер** | Распределение `@N` (sticky-логика, переиспользование, TTL) |
| **Telegram-слой** | `python-telegram-bot` + handlers: публичные / админские / inline retry |
| **Очередь ретраев** | SQLite + фоновой воркер с экспоненциальным бэк-оффом |
| **GPS / SOS / категории / whitelist** | Опциональные модули поверх ядра |
| **GUI** | PyQt6, отдельный процесс — запускает relay.py как subprocess |

`relay.py` и `gui.py` живут в **разных процессах**. GUI запускает relay.py через `QProcess`, читает stdout/stderr → раскрашенный лог. SQLite позволяет читать одну БД из обоих (короткие соединения).

## Что реализовано

### Слоты `@N` со sticky-логикой
- Каждое первое сообщение от уникального TG-юзера получает наименьший свободный `@N` (1, 2, 3, …)
- TTL до первого ответа — **20 часов**
- После ответа от Михаила слот переходит в **sticky** на **10 часов** дополнительно
- Повторные сообщения юзера — тот же `@N`, TTL обновляется
- Освобождённые `@N` переиспользуются → нумерация остаётся компактной

**Зачем sticky**: переписка должна сохранять контекст. Если юзер задал вопрос и ты ответил, его «спасибо» через 5 минут не должен ехать на новом `@N` — это и захламляет твой экран в кармане, и теряет связь между сообщениями.

### Статусы доставки + ACK
Каждое сообщение проходит цепочку:

| Статус | Когда | Источник |
|---|---|---|
| `📨 Передаю сообщение для {имя}…` | Сразу как только TG-юзер написал | Бот ставит placeholder |
| `📨 Сообщение отправлено. Ответ обычно 2–5 минут.` | `sendText()` отработал без исключения | Edit статуса |
| `✓ Доставлено: {имя}.` | LoRa-ACK от карманной ноды дошёл до home | Отдельное сообщение, `onResponse` callback |
| `{имя}: текст ответа` | Михаил написал `@N ответ` с кармана | Новое сообщение |
| `⏳ {имя} пока вне связи. Сообщение в очереди.` | sendText упал, ставим в `retry_queue` | Edit статуса + inline-кнопка «🔄 ещё раз» |
| `⌛ Связь не восстановилась — не доставлено.` | Прошёл TTL слота, ретраи не помогли | Edit статуса |

**Защита от тишины**: если `edit_message_text` не удался (Telegram иногда отказывается), хелпер `_show_status` шлёт **новое сообщение** с тем же текстом. Юзер всегда видит финальный статус.

### Авто-ретраи
Таблица `retry_queue` в SQLite. Фоновой `retry_worker` каждые 60 сек:
- Удаляет строки с истёкшим `deadline` (== TTL слота) и обновляет статус юзеру
- Для остальных пытается `send_dm_to_pocket()` ещё раз
- На неудаче — экспоненциальный бэк-офф: 2 → 4 → 8 → 16 мин (capped в `RETRY_MAX_INTERVAL_MIN`)

**Зачем в БД, не в памяти**: переживает рестарт бота. Если сервер упал в момент висящего ретрая — после старта продолжит.

### Категории / реферальные deeplinks
Таблица `categories(name, tag)`. Каждая пара превращается в готовую ссылку `https://t.me/your_bot?start=<тег>`. Когда юзер пришёл по такой ссылке:
1. `cmd_start` ловит payload в `context.args[0]`
2. Если тег есть в `categories` — пишется в `users.entry_tag`
3. В префикс LoRa-пакета добавляется метка: `[@3 work:vasya 16:15] заказ готов?`

У тебя в кармане сразу видно, **откуда** сообщение — коллеги, друзья, семья. Команда `/link` боту печатает все ссылки одним списком.

**Зачем**: один бот для разных аудиторий, без потери контекста. Раздаёшь разные ссылки разным группам — у тебя в кармане они автоматически отмаркированы.

### GPS / `/where` (бета)
Если на pocket-ноде есть GPS-модуль и `gps_enabled=True`:
- `relay.py` слушает `POSITION_APP` пакеты от карманной ноды → пишет `(lat, lon, alt, fix_time, received_at)` в БД
- Команда `/where` от **избранных** юзеров возвращает Telegram-локацию + строку «Фикс N мин назад»
- Не-избранным **отвечается одинаково**: «Локация недоступна» — нельзя по ответу понять, что команда вообще доступна
- Rate limit: 1 запрос в N минут на юзера (Михаил без лимита)
- Команды управления: `/fav <id>`, `/unfav <id>`, `/favlist`, `/gps` (отладка)
- На карманной ноде: `@N !fav` — добавить активного отправителя в избранные

**⚠ Бета**: автор не тестировал — нет физического GPS-модуля. Логика написана по спецификации Meshtastic POSITION_APP. Поведение «недоступно» для всех — в том числе когда фича выключена — намеренно: не показываем, что система настроена.

### SOS / тревога
Триггер: с карманной ноды напиши `#SOS текст`. Бот разошлёт всем из списка SOS-получателей:
- Пред-настроенный текст (по умолчанию «🆘 {имя} нужна помощь. Это автоматическое уведомление.»)
- Текст после `#SOS` от тебя
- Опционально координаты pocket-ноды (если `sos.include_coords=True` и GPS работает)

Получатели не должны ничего настраивать — они получают сообщение даже если никогда не писали боту. SOS в выключенном состоянии (`sos_enabled=False`) тихо игнорирует `#SOS` (только зеркалит тебе в TG как предупреждение «триггер был, но фича выключена»).

**Защита от ложных срабатываний**: их нет специально. Это «красная кнопка» — false positive дешевле false negative.

### Whitelist (закрытый режим)
Опциональный закрытый режим: если `whitelist_enabled=True`, бот принимает сообщения только от пользователей с пометкой WL. Управление в окне «Пользователи» или TG-командами `/allow /disallow /allowed`. Бан-лист работает поверх — забаненный не пройдёт даже из whitelist.

**Зачем**: бот публичный по дизайну (любой с ссылкой может писать), но иногда нужно временно прикрыть.

### Reply-клавиатура + inline retry
Под полем ввода у TG-юзера висят **2 persistent-кнопки**: «📍 Где {имя}» и «ℹ️ Помощь». При неудачной доставке под статусным сообщением появляется inline-кнопка «🔄 Попробовать ещё раз» — мгновенный ручной retry поверх фонового авто-ретрая.

### Команды с карманной ноды
| Команда | Что делает |
|---|---|
| `@N текст` | ответ юзеру слота N (sticky-extension TTL) |
| `@N !ban` | бан юзера слота N (тихий) |
| `@N !fav` / `@N !unfav` | избранные (доступ к /where) |
| `!status` | список активных слотов в DM-ответе |
| `!ping` | `pong slots=N up=Mh` — проверить что бот живой |
| `!history` / `!last` | последние сообщения за N часов |
| `!gps` | отладочный вывод GPS-фикса |
| `!favlist` | список избранных |
| `!help` | краткая справка |
| `#SOS текст` | тревога |
| `@ai вопрос` / `@aiN текст` | AI-помощник (если включён) |

`!history` / `!last` поддерживают аргументы: `!history` — дефолтные 8 сообщений за 5 часов; `!history 5` — 5 сообщений за дефолтные часы; `!history 3h` — за 3 часа с дефолтным лимитом; `!history 5 3h` — оба параметра. Защитные пределы 24h / 20 items. Сообщения старше `HISTORY_RETENTION_DAYS` (по умолчанию 7 дней) автоматически удаляются.

Всё через парсер `parse_mesh_text` — однозначные регэкспы по приоритетам.

### Поддержка 50 моделей нод
В `devices/` лежат SVG-файлы с офсайта Meshtastic (Heltec V3..V4, T-Beam, T-Echo, T-Deck, RAK4631, Xiao S3, ThinkNode M1-M6, и т.д.). В Settings → Устройство пользователь выбирает свою модель → правая панель главного окна показывает реальную картинку.

Авто-детект по описанию COM-порта **намеренно не делается**: VID/PID и описание драйвера разные на разных ОС/дровах, надёжно не определишь. Лучше один раз руками выбрать.

### GUI на PyQt6
- **Главное окно**: меню (Файл/Правка/Вид/Сервис/Справка), тулбар (Старт/Стоп/COM/Settings/Users/Slots/Categories), splitter (лог + правая инфо-панель с картинкой ноды и списком слотов), статус-бар
- **Settings dialog**: sidebar-навигация на 8 секций (Бот / Pocket / Устройство / Лимиты / GPS β / SOS / Категории / Whitelist)
- **Users dialog**: таблица с фильтрами WL/Fav/Ban + поиск + ручное добавление + per-row чекбоксы
- **Slots dialog**: таблица активных слотов с TTL-progress bar, текстом сообщения, sticky-меткой
- **Onboarding wizard**: 3-шаговый мастер первого запуска (токен → owner ID → pocket ID)
- **About**: карточка с логотипом, версией и ссылкой на meshgram.site
- **Тёмная тема** через QSS (палитра portирована из браузерного прототипа дизайнера)
- **Reply-клавиатура и inline-кнопки** видны в TG, не в этом окне

### Хранение конфига в `.env`
Не JSON, не TOML, не INI. Стандартный dotenv-формат: `KEY=VALUE`, `# comment`, кавычки только когда нужны. Файл — **источник истины** и для GUI, и для relay.py.

### `display_name`
В настройках есть поле «Имя получателя» — все user-facing тексты используют его (`📨 Передаю сообщение для {имя}`, `✓ Доставлено: {имя}`, кнопка «📍 Где {имя}»). По умолчанию «Михаил». Если бот деплоится для другого человека — поменял в Settings → Бот → Имя получателя, перезапустил, всё.

## Почему сделано так, а не иначе

**Почему .env, а не JSON?**
- Стандартный формат, его все знают (12-factor приложения)
- Удобно деплоить и в Docker, и через systemd
- Можно править руками без риска сломать синтаксис скобок
- Свой парсер на 30 строк, не таскаем `python-dotenv` зависимостью
- `.env.example` — конвенциональный паттерн для шаблона

**Почему DM, а не broadcast?**
- Privacy: чужие ноды на канале видят только зашифрованный блоб
- Адресность: ACK работает (от broadcast его не получишь)
- Простота: home → pocket идёт ровно одним адресом

**Почему слоты `@N`, а не TG username в префиксе?**
- LoRa-пакет ~200 байт, длинные имена не влезают
- `@3` — 2 символа, легко набрать на ноде
- Sticky-логика становится тривиальной

**Почему subprocess (QProcess) вместо in-process?**
- GUI можно закрыть — relay.py продолжает работать
- Перезапуск GUI не убивает бота
- stdout/stderr легко стримить в окно
- Падение GUI не валит TG-цикл

**Почему PyQt6, а не Electron / Tauri / web?**
- Нативное Windows-приложение, без JS-тулчейна
- Один Python-стек на всё
- Для конечного юзера: один .bat, не нужен Node/npm
- Минус: анимации в Qt беднее CSS, glow-эффекты лоссово

**Почему SQLite, а не файлы?**
- Concurrent reads из gui.py и relay.py одновременно — атомарно из коробки
- Транзакции для retry-очереди — без подсчёта блокировок
- Миграции через `ALTER TABLE` без рисков потерять данные

**Почему retry_queue в БД, не в памяти?**
- Переживает рестарт сервера / бота
- Если упал в момент очередного ретрая — продолжит после старта
- Видно всю очередь в SQL

**Почему sticky-слоты на 10ч (а не Telegram-style вечно)?**
- LoRa-протокол не безразмерный — слоты должны переиспользоваться
- 10ч после последнего ответа = окно для естественной переписки
- Можно поменять в Settings → Лимиты

**Почему ACK через `wantAck=True` сейчас, а не fire-and-forget?**
- Позволяет показать «✓ Доставлено» — реальную информацию для юзера
- Прошивка делает ретраймы автоматически — повышает надёжность
- **Минус**: при слабом канале добавляет 10-30 сек к user-perceived задержке. **Возможно вынесем как опцию «приоритет: скорость / надёжность»** в будущем

**Почему авто-детект модели ноды не делаем?**
- VID/PID разный на разных платформах
- Описание драйвера не стандартизировано
- Один раз руками = надёжнее болтанки
- Дополнительный плюс: явный выбор → можно показать правильную картинку

**Почему `display_name` без склонения по падежам?**
- Русское склонение требует знания пола получателя и формы → быстро сложно
- Все фразы переписаны в нейтральную форму («Передаю сообщение для X», «Доставлено: X», «Связь с X не восстановилась»)
- Работает для любого имени, мужского или женского

**Почему GPS заблокирован за `gps_enabled` флагом по умолчанию?**
- Автор не тестировал на железе
- Лучше не «как бы работающая» фича в проде, а явное «бета, включи если хочешь»
- Для не-избранных и при выключенной фиче ответ одинаковый: «Локация недоступна» — нельзя понять, что команда вообще существует

**Почему `#SOS` не имеет защиты от ложного срабатывания?**
- Это «красная кнопка». False positive (случайно крикнул SOS) → пара извинений в чате. False negative (не сработало в момент истины) → катастрофа. Преимущество в пользу простоты и надёжности срабатывания

## Что планируется

### Карта в GUI
Сейчас GPS показывается только текстом «N минут назад / координаты». Хочется встроить карту в правую панель главного окна (или в отдельный диалог): пятно с положением карманной ноды, опционально zone-coverage по последним hops. Технически — `QWebEngineView` + Leaflet/OpenStreetMap. Дизайнер карту не нарисовал, но Qt умеет встроить браузерный движок.

### Команды управления нодой
В вкладке Settings → Устройство есть заглушка «Команды устройства». В планах: **повышение/понижение мощности LoRa, переключение роли (client / router), смена региона/канала, рестарт ноды** прямо из панели. Через `meshtastic.localNode.*` и AdminMessage. Делать аккуратно — неудачная команда может «закирпичить» настройку.

### Свои ноды-ретрансляторы по городу
Поставить 3-5 стационарных нод на крышах/балконах в ключевых точках. **Без Python**: в прошивке Meshtastic выставить `role = ROUTER`, объединить общим приватным каналом — прошивка сама делает multi-hop. Со стороны `relay.py` добавить только мониторинг этих нод (видна / не видна, lastHeard) и индикатор в GUI «связь по городу: 4/4 ✓».

### Wi-Fi / TCP подключение к ноде
Сейчас home-нода обязательно по USB. Meshtastic умеет `tcp_interface.TCPInterface(hostname=...)` — нода в локалке по IP. Полезно если USB-кабель неудобен или нода питается стационарно. Развилка в `_connect_mesh`, в настройках поля `CONNECTION_TYPE=serial|tcp` и `TCP_HOST`.

### Опция «приоритет: скорость / надёжность»
Тоггл в настройках:
- **Надёжность** (текущий режим): `wantAck=True`, ретраи прошивки, «✓ Доставлено» — но 10-30 сек дополнительной задержки
- **Скорость**: `wantAck=False`, fire-and-forget — задержка как у мобильного приложения Meshtastic, но «✓ Доставлено» убирается, остаётся только «📨 Отправлено»

### Auto-restart при сетевых сбоях ✅ (v0.5.1+)
`telegram.error.NetworkError` / `TimedOut` / `RetryAfter` — `_run_polling_with_retry` ловит и переподключается с exp-backoff (5s → 10s → ... → 5min cap). Если стабильно проработали 10+ минут — backoff сбрасывается. Не падает на типичных кратких обрывах интернета.

### Auto-recovery при потере USB-связи с нодой ✅ (v0.8.0+)
Два механизма:
1. **`meshtastic.connection.lost` pubsub-event** — meshtastic-python публикует это событие при разрыве. Подписчик `_on_mesh_connection_lost` валит процесс (`os._exit(1)`) — supervisor рестартует.
2. **`mesh_watchdog` background-task** — раз в 30 секунд проверяет что `_mesh_iface.myInfo` не None. Если 3 раза подряд миссинг (90+ секунд тишины) — тоже жёсткий exit. Это safety-net на случай «hard» отрыва когда event не пришёл.

В GUI (`gui.py`) — `_on_finished` callback автоматически рестартует relay через 5 секунд при exit-code != 0. Burst-protection: не больше 3 рестартов за 5 минут — после этого нужен ручной разбор. Юзер не должен сам кликать «Старт» при USB-плагине обратно.

### Сайт `meshgram.site`
Куплен, пуст. План: лендинг + публичная документация + список совместимых железок (со страницы каталога моделей в этом проекте). Стек — пока не решено, скорее всего что-то лёгкое (Astro, Tailwind).

### Анимации UI
Текущая Qt-реализация имеет минимум анимаций. Дизайнер прислал прототип с pulse-точками, scan-line, log-fade-in, dialog spring-in, прогресс-шиммер. В Qt всё это делается через `QPropertyAnimation` + кастомный paint — займёт пару дней работы. Пока сделано только базовое (пульс GlowDot).

### Дозалив дизайнерских картинок Meshtastic
Сейчас в каталоге 50 SVG. На офсайте Meshtastic регулярно появляются новые модели. Возможно сделать скрипт, скачивающий обновления с их репозитория автоматически.

## Безопасность и приватность

- **Контент**: home ↔ pocket идёт DM-ом, end-to-end шифрован Meshtastic-протоколом. Чужие ноды на канале не читают
- **Метаданные**: чужие ноды видят, что между двумя твоими нодами идёт зашифрованный трафик. Чтобы скрыть и это — заведи отдельный приватный канал в приложении Meshtastic (custom name + рандомный PSK), оба своих устройства подключи к нему. Это настройка железа, не кода
- **Токен**: `.env` в `.gitignore`. В репо идёт только `.env.example`
- **Бан**: тихий — пользователь видит «не принимает сообщения», не узнаёт что забанен после конкретного сообщения
- **Whitelist**: опциональный закрытый режим
- **GPS-избранные**: не-избранные получают тот же ответ что и при выключенной фиче — нельзя понять, что команда `/where` вообще существует

## Ограничения

- **170 символов** на сообщение (LoRa-пакет ~200 байт минус префикс ~30). Кириллица в UTF-8 = 2 байта/символ → длинный русский текст может не уместиться, прошивка дробит на пакеты со снижением надёжности
- **Только текст**. LoRa не для медиа
- **LoRa packet loss** даже в зоне приёма. `wantAck=True` включает ретраи прошивки, но 100% доставку гарантировать нельзя
- **Telegram NetworkError** при сетевых сбоях роняет процесс. Перезапусти. На будущее — auto-restart wrapper
